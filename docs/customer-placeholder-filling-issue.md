# 占位客户填报问题记录

> 记录日期:2026-07-23
> 来源:客户名称归属解析修复(backend/src/services/dingtalk.ts `findBefore`)后的数据验证
> 性质:**源数据(钉钉表单填报)问题**,不是系统解析问题。系统忠实反映表单数据,不在系统侧加规则处理。

## 现象

钉钉审批表单中的「客户名称」字段是必填的 CRM 关联字段(OpenDataField / TableField)。
大量员工在**非真实客户拜访**的行程中,随手选择占位客户凑数,导致客户名称与实际拜访内容不匹配。

常见的三条占位客户记录(全公司共用的"假客户"):

| 占位客户 | 典型使用场景 |
|---|---|
| `公司（签到用）` | 去公司办公、开会 |
| `虚拟客户（签到用）` | 代理商相关、不便选真实客户时 |
| `住址（签到用）` | 回家、或纯粹敷衍填写 |

系统解析结果完全忠实于表单:员工选什么,控制台就显示什么。因此这不是解析 bug,而是填报质量问题,预计波及约 90% 的外勤人员。

## 证据 1:李朝晖 2026-07-16(审批单 202607160730000045200)

当天实际行程:07:30 从家(东莞荣昌街)出发 → 08:32 到深圳创维数字大厦(**公司办公**)→ 19:46 回家。

- 表单「客户名称1」(出发前申报的第一个目的地)= `住址（签到用）`
- 表单「客户名称2」(第二个目的地,回家)= `住址（签到用）`
- 两个字段指向同一条 CRM 记录(relateId 完全一致),确为员工所选,非系统复制。

矛盾点:去创维是"公司办公",客户名却挂了"住址(签到用)"。

### 填报习惯证据(该员工最近 20 天第一个拜访点的客户名)

| 拜访类型 | 通常选的客户 |
|---|---|
| 真实客户拜访 | 真实客户名(欣源机械、达意隆、锐欧仕、东福五金…) |
| 公司办公/开会 | `公司（签到用）`(7-13、7-10、7-01 均如此) |
| 代理商相关 | `虚拟客户（签到用）` |
| 敷衍填写(拜访情况写"1") | `住址（签到用）` |

7-16 属于"公司办公",按他自己的习惯应选 `公司（签到用）`,实际却选了 `住址（签到用）`,偏离其个人常态。

最可能的原因:

1. **复制前一天的审批单提交**:钉钉审批"再次提交"会带出上次字段值,他 7-15 的客户名称1 恰好就是 `住址（签到用）`;
2. 或手滑选了 CRM 最近使用列表第一条——每天回家都用"住址(签到用)",它永远排在最前。

## 证据 2:范德康 2026-07-01(审批单 202607010902000523683,正面案例)

- 表单「客户名称1」= `济南禾瑞机械设备有限公司`,CRM 记录地址为"济南市历下区山大路157号**华强广场**E-2909室";
- 实际签到点:09:48 山大路155号靠近**华强广场**,拜访情况"拜访代理商"。

CRM 客户地址与实际签到点完全吻合,说明**认真填报时该字段是准确、可用的**;占位问题是态度/流程问题,不是字段设计完全失效。

## 原因分析

1. 客户名称为必填项,而"公司办公""回家"等场景在客户库中没有合适的真实客户可选;
2. 公司客户库里存在 `公司（签到用）`、`虚拟客户（签到用）`、`住址（签到用）` 三条占位记录,给敷衍填写提供了便利;
3. 钉钉"再次提交/复制表单"功能会带出前一天的值,不修改就直接提交;
4. CRM 关联字段按"最近使用"排序,占位客户因高频使用永远排在最前。

## 影响

- 控制台/报表中按客户维度统计时,占位客户会污染数据(如"住址(签到用)"看起来像一个被频繁拜访的客户);
- 单独看某条记录时,客户名不可直接采信,需结合「本次拜访情况」判断。

## 结论与建议

- **不定性为系统 bug,不在分析系统侧加校验规则**;
- 这是源数据治理问题,可选的源头处理方式(供管理决策):
  1. 表单层面:客户名称改为非必填,或增加"无客户(公司办公/回家)"之类的明确选项,替代占位客户;
  2. CRM 层面:清理/禁用三条占位客户记录,或限制其可选范围;
  3. 管理层面:将"占位客户与拜访类型不匹配"纳入日常抽查口径,用本文档数据作为对比证据。
