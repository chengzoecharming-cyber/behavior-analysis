# 销售外勤行为分析系统 - 开发计划

> 最后更新：2026-07-14
> 当前重点：一期反馈需求——统一查询页 Dashboard 下钻、组织排行榜行内展开、部门数据治理（等 HR 输入）。

## P0 热修复（当前冲刺）

### 背景

接入真实数据后发现，系统存在两类阻断性风险：

1. **时间轴语义不统一**：Excel 日期解析异常、部分代码未按北京时间处理业务日期、前端日期选择器受浏览器时区影响，导致"选 7 月 1 日查不到数据"、"风险摘要算错日期"、"工作日/周末误判"。
2. **权限系统未收口**：大量核心业务接口未挂载认证与角色过滤，存在数据泄露和越权操作风险。

这两类问题不解决，后续所有分析功能都建立在不可信的数据和不受控的访问之上。

### P0-1 时间轴语义统一

#### 产品目标

> 定义并落地「业务时间契约」：**任何无显式时区的日期/时间字符串，统一按 Asia/Shanghai（UTC+8）解释；数据库继续以 UTC 时间戳存储；前端日期选择器反映业务日期，而非浏览器本地日期。**

#### 核心原则

| 原则 | 说明 |
|---|---|
| 业务日期唯一解释 | `2026-07-01` = 2026-07-01 00:00:00 +08:00，不受服务器/浏览器时区影响 |
| 数据库统一存 UTC | 使用 `TIMESTAMPTZ`，写入前完成 +08:00 → UTC 转换 |
| 前后端参数简化 | 前端传 `YYYY-MM-DD`，后端按 +08:00 解释 |
| 工作日按北京时间 | 周末/工作日判断基于 Asia/Shanghai 的星期几 |

### P0-3 控制台日期下钻状态修复（已修复）

#### 问题

个人周期总览图表点击某日（如 6 月 18 日）后，新标签页没有正确加载该日期数据，而是跳转到了该员工最近有数据的日期（如 7 月 13 日）。

业务方一度怀疑是「这一天该员工属于别的部门」导致数据被过滤，但实际原因是 `ConsolePage.tsx` 中「获取可用日期列表」与「设置 selectedDate」放在同一个 `useEffect` 里，异步返回时闭包中的 `dateRange` 已经过期。

#### 修复

- 将单一 `useEffect` 拆分为两个：
  1. 仅在 `scope` / `userId` 变化时获取可用日期列表。
  2. 在 `dateRange` 或可用日期列表变化时，根据最新 `dateRange` 设置 `selectedDate`。
- 文件：`frontend/src/pages/ConsolePage.tsx`
- 已验证：通过浏览器自动化打开 `/console?user=282454695925857408&date=2026-06-18`，页面正确选中 6 月 18 日并加载当日轨迹与异常数据。

## 一期反馈需求

> 来源：一期功能上线后业务方反馈收集与产品经理沟通确认。
> 核心原则：**尽量少做新功能，尽量复用现有能力；不重新调用高德 API，以数据聚合计算为主。**

### 背景与定位

一期上线后暴露出两个核心问题：

1. **入口分散**：控制台、日历视图、周期总览、决策系统各自为战，用户需要在多个页面间跳转。
2. **视角混乱**：团队 / 个人 / 区域等概念未统一，导致地图、排行榜、轨迹等内容的展示方式不一致。

结论：**把项目当作 Dashboard + 查询工具来设计**。Dashboard 负责总览和下钻入口；查询页负责承载所有查询视角。

### 产品架构决策

#### 1. 页面结构：Dashboard + 统一查询页

| 页面 | 职责 | 内容 |
|---|---|---|
| **Dashboard（决策系统首页）** | 总览 + 下钻入口 | 核心指标卡、趋势图、Top 异常、快捷入口 |
| **查询页（原 Console / 日历视图 / 周期总览）** | 所有查询与分析 | 一个页面，通过级联选择器和日期选择器切换查询范围与视图 |

**注意**：CalendarView 与 PeriodOverview 不再作为独立页面存在，它们是查询页在个人层级下不同日期模式的表现。

#### 2. 查询范围：四级级联选择

基于钉钉通讯录的部门树，查询范围四级级联：

```
全公司 > 部门 > 子部门 > 个人
```

- **"子部门"不是独立实体**，它就是部门树中的下一级节点；前端不会展示"子部门"这种字段，只显示真实部门名称。
- 组织架构直接走钉钉通讯录同步，当前只有销售部等部门划分了子部门，但系统层面统一按树形处理。
- 父级查询结果支持下钻到子级，**另开标签页**打开并自动带入对应筛选条件。

#### 3. 单选互斥规则

| 选择层级 | 说明 |
|---|---|
| 全公司 | 部门、人员选择器不可用 |
| 部门 | 只能选一个部门，人员选择器不可用；可从结果下钻到子部门或个人 |
| 子部门 | 同部门逻辑，范围限定在该子部门 |
| 个人 | 只能选一个人，部门选择器不可用 |

#### 4. 日期选择器

使用统一的日期范围选择器，**本身同时支持单日和范围**，无需额外的单日 / 区间切换开关。

#### 5. 视图内容

| 查询范围 | 日期模式 | 展示内容 |
|---|---|---|
| 全公司 / 部门 / 子部门 | 单日或区间 | 指标卡 + 热力图 + 排行榜 + 趋势图 + 里程对比图 |
| 个人 | 单日 | 日历视图 / 单日轨迹 / 异常清单 |
| 个人 | 区间 | 周期总览（趋势、拜访统计、里程对比等） |

**排行榜**：不需要维度切换，就是一个表格，用户通过点击表头字段排序。

**趋势图**：复用当前决策台的图表设计，展示拜访次数、里程情况。

**热力图**：在公司 / 部门 / 子部门层级均展示，只是数据范围不同。

**个人轨迹**：仅在个人单日模式下展示，避免多人轨迹叠加成"毛线团"。

#### 6. 行为报告

每一层级都可以生成对应范围的行为报告：

- 全公司行为报告
- 部门行为报告
- 子部门行为报告
- 个人周期报告（已有功能保留）

**发送方式**：暂定统一发送到当前群聊，个人单独发送延后处理。

#### 7. 权限与数据范围

| 角色 | 可查询范围 |
|---|---|
| admin | 全公司 / 任意部门 / 任意个人 |
| manager | 自己部门及其子部门 / 自己团队成员 |
| staff | 仅个人 |

#### 3.5.6 里程累计值口径与地图按审批隔离

> 发现时间：2026-07-02

##### 问题

接入真实钉钉审批数据后发现，`visits.reported_distance_km` 被直接 `SUM` 作为员工填报总里程，但钉钉表单里的 `累计里程N` 实际是**截至本次签到的累计值**（后一段包含前一段），导致同一审批内的多次签到被重复累加，偏差可达数倍。

数据验证（311 条有里程的审批单）：

| 校验项 | 结果 |
|---|---|
| 最后一个累计值 = 今日累计里程 | 300 / 311（96.5%） |
| 各段累计值直接相加 = 今日累计里程 | 76 / 311（24.4%） |
| 累计值与里程表读数（终点 − 起点）一致 | 272 / 311（87.5%） |
| 相邻累计值之差 = 相邻终点读数之差 | 276 / 311（88.7%） |

结论：**应取每个审批单的最后一个累计值作为该次出差总里程**，同时清洗负值、>5000 km 等异常读数。

##### 方案

**后端口径修正：**
- `/analytics/mileage` 按 `approval_id` 分组，每组取 `MAX(reported_distance_km)`，过滤异常值后再跨组求和。
- `routeService` 计算路线时按 `approval_id` 分组，组内相邻点连线，**不同审批单之间不连线**。
- `mileageAnalysis` 分段里程优先使用里程表读数差（`curr.end_odometer − prev.end_odometer`），补全首段计算。
- 导入时校验：若 `end_odometer < start_odometer` 或单次差值 > 5000 km，标记为异常读数，不写入 `reported_distance_km`。
- 历史数据：清空 `routes` 表，按新规则重算；刷新 `risk_summary_cache` 与 `anomalies`。

**前端展示调整：**
- 控制台/地图在同一天、同一员工存在多个 `approval_id` 时，需要按审批单拆分展示（如 Tab、下拉选择或卡片列表），避免把多个独立出差串成一条轨迹。
- 单个审批单内仍按时间顺序展示完整轨迹；跨审批的“汇总视图”可作为可选功能，但需明确标注“多行程合并”。

CREATE TABLE vehicles (
  id SERIAL PRIMARY KEY,
  plate VARCHAR(32),

  user_id INTEGER REFERENCES users(id),

);

  id SERIAL PRIMARY KEY,
